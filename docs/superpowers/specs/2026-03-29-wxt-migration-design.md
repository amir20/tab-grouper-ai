# WXT Migration Design Spec

## Goal

Migrate the Gruper browser extension from `vite-plugin-web-extension` to WXT framework to enable multi-browser builds (Chrome + Firefox) while preserving all existing functionality.

## Project Structure

Current flat `src/` layout migrates to WXT conventions:

```
entrypoints/
  background.ts              # Service worker (Chrome) / background script (Firefox)
  popup/
    index.html
    main.ts                  # Vue app bootstrap
    App.vue                  # Main popup component
    style.css                # Tailwind + theme tokens
components/
  GroupItem.vue
  ModelPicker.vue
  ProgressBar.vue
  StatusDot.vue
composables/
  useEngine.ts               # Engine lifecycle, provider switching, tab grouping
utils/
  config.ts                  # Types, constants, Chrome storage helpers, tab logic
  openrouter.ts              # OpenRouter API client
assets/
  system-prompt.txt          # LLM system prompt
public/
  icons/                     # Extension icons (unchanged)
wxt.config.ts                # Replaces vite.config.ts + manifest.json
```

WXT auto-discovers entry points in `entrypoints/` and generates the manifest.

## Manifest Generation

`wxt.config.ts` defines:

- **Shared fields**: name, version, description, permissions (`tabs`, `tabGroups`, `storage`, `notifications`), host_permissions (HuggingFace, OpenRouter), CSP (`wasm-unsafe-eval`), commands (Alt+Shift+G), icons
- **Entry point metadata**: WXT infers popup and background config from file conventions
- **Vite plugins**: `@tailwindcss/vite` and `@vitejs/plugin-vue` configured in `wxt.config.ts`'s `vite` option
- **Build settings**: `assetsInlineLimit: 0` and `chunkSizeWarningLimit: 6000` for WebLLM WASM

## Browser Targeting

- `wxt dev` — Chrome dev with HMR
- `wxt dev --browser firefox` — Firefox dev
- `wxt build` — Chrome production build
- `wxt build --browser firefox` — Firefox production build
- `wxt zip` — Zip for both browsers

## WebLLM on Firefox

No build-time feature gating. Runtime approach:

1. `useEngine.ts` attempts to create the WebLLM engine regardless of browser
2. If creation fails (WebGPU unavailable, service worker limitations), catch the error
3. Set a user-visible message: "Local AI unavailable in this browser. Switch to OpenRouter for cloud-based grouping."
4. User can switch to OpenRouter from the Model picker

## API Compatibility

WXT provides a `browser` namespace polyfill. However, we'll keep using `chrome.*` directly since:
- WXT makes `chrome.*` available in both Chrome and Firefox builds
- Avoids unnecessary refactoring of every API call
- All `chrome.tabs`, `chrome.tabGroups`, `chrome.storage`, `chrome.runtime` calls remain as-is

## Files Removed

- `manifest.json` — generated by WXT
- `vite.config.ts` — replaced by `wxt.config.ts`
- `src/popup.html` — WXT generates popup HTML
- `src/env.d.ts` — WXT provides its own type declarations
- `src/popup.ts` — becomes `entrypoints/popup/main.ts`
- `src/service-worker.ts` — becomes `entrypoints/background.ts`

## Dependencies

### Added
- `wxt` (replaces `vite-plugin-web-extension` and direct `vite` usage)

### Removed
- `vite-plugin-web-extension`
- `vite` (WXT bundles its own)
- `@types/chrome` (WXT provides types)

### Kept
- `@mlc-ai/web-llm`
- `vue`
- `@tailwindcss/vite`
- `@vitejs/plugin-vue`
- `tailwindcss`
- `typescript`

## What Stays the Same

- All Vue component markup and logic
- `useEngine.ts` composable logic
- `config.ts` utility functions and types
- `openrouter.ts` API client
- `system-prompt.txt` content
- Tailwind v4 theme and styles
- Public icons
